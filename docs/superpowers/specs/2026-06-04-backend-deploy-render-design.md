# SP26 — Backend Deploy to Render (design)

Date: 2026-06-04

## Goal

Give the Halal Checker API a stable public URL so the mobile app no longer
depends on the dev machine + LAN/tunnel. Target platform: **Render**, deployed
as a Docker web service with a managed **PostgreSQL** database, with **Tesseract
OCR** included so `/scan-image` works in production. Infrastructure is described
as code in a committed `render.yaml` Blueprint.

Non-goals: Alembic migrations (the existing `create_all` is sufficient for a
fresh Postgres database), CI/CD auto-deploy pipelines beyond Render's built-in
git auto-deploy, a custom domain, and the Redis shared rate limiter (MED-1,
still deferred until a multi-worker deploy).

## Decisions (locked during brainstorming)

- **Platform:** Render.
- **Database:** Render managed PostgreSQL (data persists across deploys).
- **OCR:** Included — Docker image installs `tesseract-ocr`.
- **Deploy mechanism:** `render.yaml` Blueprint (infrastructure as code, committed).

## Components

### 1. Postgres driver support (code change)

The repo currently only ships a SQLite path. Two small changes make it speak
Postgres:

**`pyproject.toml`** — new optional dependency group:
```toml
postgres = ["psycopg[binary]>=3.1"]
```
psycopg3 with bundled binaries. Kept optional so local SQLite installs stay
lean; the Docker image installs `.[ocr,postgres]`.

**`src/halal_scanner/db.py`** — normalize the URL inside `_make_engine()` before
`create_engine`. Render provides `postgresql://…` (and Heroku-style
`postgres://…` is common); SQLAlchemy would otherwise default to the
uninstalled psycopg2 driver. Map:

| Input scheme        | Normalized to                |
|---------------------|------------------------------|
| `postgres://…`      | `postgresql://…`             |
| `postgresql://…`    | `postgresql+psycopg://…`     |
| `sqlite…` / other   | unchanged                    |

The two rules apply in order (`postgres://` → `postgresql://` →
`postgresql+psycopg://`). A URL that already names a driver (e.g.
`postgresql+psycopg2://`) is left as-is. This is pure string logic and is
unit-tested (TDD).

No Alembic — `Base.metadata.create_all(bind=engine)` (called at import time in
`app.py`) creates all tables on first boot against the empty Postgres database.

### 2. `Dockerfile`

```dockerfile
FROM python:3.11-slim

RUN apt-get update \
 && apt-get install -y --no-install-recommends tesseract-ocr \
 && rm -rf /var/lib/apt/lists/*

WORKDIR /app
COPY pyproject.toml ./
COPY src ./src
RUN pip install --no-cache-dir ".[ocr,postgres]"

ENV PORT=8000
CMD ["sh", "-c", "uvicorn halal_scanner.api.app:app --host 0.0.0.0 --port ${PORT}"]
```

- `tesseract-ocr` provides the `eng`+`osd` data and a `tesseract` binary on
  PATH. `ocr.py`'s `_resolve_tesseract_cmd` finds it on PATH automatically on
  Linux — no `HALAL_TESSERACT_CMD` needed.
- Binds `0.0.0.0:$PORT`; Render injects `$PORT`. Default `8000` for local
  `docker run`.

### 3. `.dockerignore`

Exclude from the build context (smaller image, no local secrets/DB leakage):
`mobile/`, `tests/`, `docs/`, `.venv/`, `*.db`, `.git/`, `__pycache__/`,
`QA_SECURITY_FINDINGS.txt`, `examples/`.

### 4. `render.yaml` Blueprint

```yaml
databases:
  - name: halal-db
    plan: free
    databaseName: halal_scanner

services:
  - type: web
    name: halal-scanner-api
    runtime: docker
    plan: free
    healthCheckPath: /health
    envVars:
      - key: HALAL_DATABASE_URL
        fromDatabase:
          name: halal-db
          property: connectionString
      - key: HALAL_JWT_SECRET
        generateValue: true
      - key: HALAL_ENV
        value: production
      - key: HALAL_RATE_LIMIT
        value: "60"
      - key: HALAL_RATE_WINDOW
        value: "60"
      - key: HALAL_TRUST_PROXY
        value: "1"
      - key: HALAL_ADMIN_EMAILS
        sync: false
      - key: HALAL_CORS_ORIGINS
        sync: false
```

Rationale:
- `HALAL_ENV=production` hides the interactive docs (`/docs`, `/redoc`,
  `/openapi.json`) and forces a positive `HALAL_RATE_LIMIT` — which is set to
  `60`, so the existing prod-posture guard passes.
- `HALAL_JWT_SECRET` is generated by Render (random per service), never
  committed.
- `HALAL_TRUST_PROXY=1` keys the rate limiter on the client IP via Render's
  `X-Forwarded-For` rather than collapsing every request onto the proxy's
  socket IP (one shared bucket). **Caveat:** the limiter reads the left-most
  `X-Forwarded-For` entry, which a client could spoof; acceptable for this
  hobby deploy (scanning endpoints still require a valid API key), revisit if
  abuse appears.
- `HALAL_ADMIN_EMAILS` and `HALAL_CORS_ORIGINS` use `sync: false` so they are
  filled in on the dashboard, not committed. Admin email = the operator's
  account; CORS only needed to test an Expo **web** build against prod (native
  Expo Go uses plain `fetch`, no CORS preflight).
- The database connection string is wired from the managed Postgres instance
  via `fromDatabase`; the normalization in `db.py` adapts the scheme to
  psycopg3 at runtime.

## Data flow

```
mobile app ──HTTPS──> Render edge (TLS, sets X-Forwarded-For)
                          │
                          ▼
                  uvicorn :$PORT (Docker)  ── tesseract (OCR /scan-image)
                          │
                          ▼
                  managed PostgreSQL (accounts, keys, history, audit)
```

App boot: import-time guards (`HALAL_JWT_SECRET` present, prod posture →
positive rate limit) run, then `create_all` ensures the schema, then the service
is ready and Render's health check hits `/health`.

## Error handling / edge cases

- **Missing driver:** if `.[postgres]` is not installed but a `postgresql://`
  URL is given, SQLAlchemy raises at engine creation — surfaced loudly at boot,
  not silently. The Docker image always installs it.
- **Prod posture failure:** if `HALAL_RATE_LIMIT` is somehow non-positive, the
  app refuses to start (existing guard) — fail-closed, visible in Render logs.
- **Health check before DB ready:** `create_all` runs at import before uvicorn
  serves, so `/health` only answers once tables exist.
- **Cold start:** free tier spins down after idle; first request after idle is
  slow. Documented, not coded around.

## Testing

- **TDD unit tests** for the URL normalization in `db.py`
  (`postgres://`, `postgresql://`, already-driver'd, and `sqlite` pass-through).
- Full existing backend suite (188 passing) must stay green — the change is
  additive and SQLite behaviour is unchanged.
- `Dockerfile`/`render.yaml` are config; validated by a successful Render build
  + boot (operator step, not unit-tested). Optionally a local
  `docker build` smoke test.
